<objective>
Extend the Phase 1 engine API with a `shipments_received_history` tuple on `StationState` and a `last_shipment_received: int` field on `StationView` (and the `RetailerView` subclass), so the per-turn play view (Plan 03) can render "Last shipment received" verbatim per PLAY-01 without deriving it from indirect history slices.

Purpose: PLAY-01 explicitly requires the player to see "last week's shipments received." Phase 1's `StationView` only exposes `inventory`, `backlog`, `supply_line`, `last_order_received`, `recent_orders_received` â€” it does NOT expose the shipments-received scalar. Deriving it in the view layer is fragile (Phase 1's history records `shipments_sent_history`, not `shipments_received_history`). A small additive engine change is cleaner, keeps view code dumb, and strictly improves API symmetry between sent and received history.

Output: An updated engine that records and exposes shipments-received per station, with a new targeted test and zero regressions in the 44/44 Phase 1 test suite.
</objective>

Modify `beergame/engine/tick.py`:

6. In `receive_shipments` (step 1): when consuming `arriving = s.incoming_shipments[0] if s.incoming_shipments else 0`, stash that value in the transient: pass `_pending_shipment_received=arriving` to `replace(s, ...)`. Do NOT append to the history here â€” history appends are step 3's job (mirrors how `_demand_to_fill` and `_shipped_this_tick` are stashed in step 2 and consumed in step 3).
7. In `record_state` (step 3): append `s._pending_shipment_received` to `shipments_received_history`, and zero the transient in the same `replace(...)` call. Pattern mirrors the existing `orders_received_history=s.orders_received_history + (s._demand_to_fill,), _demand_to_fill=0` lines.
8. In `advance_pipelines` (step 5): no change needed â€” the `_pending_shipment_received` transient was already zeroed in step 3.

**Why this transient-then-record split:** Step 1 consumes the arrival but step 3 owns history appends. Routing through a transient keeps the canonical "all histories grow exactly once per tick, in step 3" invariant intact. This is exactly the pattern Phase 1 already uses for `_demand_to_fill` and `_shipped_this_tick`.

**Constraints â€” DO NOT VIOLATE:**
- Engine remains pure-Python â€” NO `import streamlit` anywhere. The AST guard in `tests/test_no_streamlit_import.py` will catch it.
- All five steps still produce a frozen new state via `dataclasses.replace`. No in-place mutation.
- Equality semantics: the new transient is `compare=False` so it doesn't break `tests/test_determinism.py`'s byte-identical state assertions.
- Backward compatible: existing callers of `build_station_view`, `StationView`, `RetailerView` continue to work. The Sterman agent doesn't read `last_shipment_received` so AI behavior is byte-identical.
