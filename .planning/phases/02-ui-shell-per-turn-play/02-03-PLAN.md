1. **Read the player's view via the engine projection** — the ONLY allowed read path (PLAY-03):
   ```python
   role = state.player_role
   view = build_station_view(state, role)
   ```
   DO NOT read `state.stations[other_index]` for any other station. DO NOT read `state.customer_demand_history` (that's information visibility violation for non-Retailer players, AND it leaks future weeks generally). The view object is the ONLY contract.

2. **Title bar (PLAY-04)**: Display week and role:
   ```python
   st.title(f":beer_mug: Week {state.week + 1} of {state.total_weeks} — You are the {role.name.title()}")
   ```
   `state.week + 1` because `state.week` is 0-indexed (Pitfall 6). First rendered week shows "Week 1 of 36"; before the player submits week 36, the screen shows "Week 36 of 36"; after submitting week 36, app.py flips phase to "done" and `play.render` is not called for week 37 (debrief takes over instead — that's PLAY-05).

3. **Metrics row (PLAY-01)** — two columns of four metrics covering inventory, backlog, last shipment received, last order received. Add a fifth row for supply_line (per the orchestrator's decision: "Show player's own supply_line mid-game — APPROVED"):
   ```python
   col1, col2 = st.columns(2)
   with col1:
       st.metric("Inventory on hand", view.inventory)
       st.metric("Backlog", view.backlog)
   with col2:
       st.metric("Last shipment received", view.last_shipment_received)
       st.metric("Last order from downstream", view.last_order_received)

   st.metric("Units in transit to you (supply line)", view.supply_line)
   ```
   `view.last_shipment_received` comes from Plan 01's engine extension. `view.supply_line` is sum of incoming_shipments — info the player owns (their own pipeline). All five values are LOCAL to the player's station — no information leak.

4. **Order-history mini-chart (PLAY-01)** — plot the player's own placed orders only:
   ```python
   me = state.stations[role.value]
   if me.orders_placed_history:
       fig = go.Figure()
       weeks = list(range(1, len(me.orders_placed_history) + 1))
       fig.add_trace(go.Scatter(
           x=weeks,
           y=list(me.orders_placed_history),
           mode="lines+markers",
           name="Your orders",
       ))
       fig.update_layout(
           height=200,
           margin=dict(l=10, r=10, t=10, b=10),
           xaxis_title="Week",
           yaxis_title="Order qty",
           showlegend=False,
       )
       st.plotly_chart(fig, key="player_order_history", width="stretch")
   else:
       st.caption("Your order-history chart will appear after your first order.")
   ```
   **Critical:** Use `width="stretch"` — NOT `use_container_width=True` (deprecated and removed-after-grace in Streamlit 1.57.0; Pitfall 4). Pass `key="player_order_history"` for stable identity across reruns (Pitfall 3 mitigates flicker). The x-axis only extends to weeks the player has actually played — `len(orders_placed_history)` — NEVER hard-coded to 36 (Pitfall 18; would imply future weeks).

   Yes, reading `state.stations[role.value]` here is permitted — it's the player's OWN station, and the `orders_placed_history` is intrinsically the player's own data. The view-only-via-build_station_view rule applies to OTHER stations' data, not the player's own.

5. **Order form (PLAY-02)** — `st.form` with `st.number_input` + `st.form_submit_button`:
   ```python
   with st.form("turn_form", clear_on_submit=True):
       st.number_input(
           "Your order this week",
           min_value=0,
           step=1,
           value=4,
           key="order_input",   # CRITICAL: app.py::submit_order reads via this key
       )
       st.form_submit_button("Advance week", on_click=on_submit)
   ```
   - `min_value=0` blocks negatives at the UI (Pitfall 5).
   - `step=1` blocks floats (Pitfall 5).
   - NO `max_value` — the canonical bullwhip can drive Factory orders to 30-80+; capping silently truncates (anti-feature AF-4 in research).
   - `value=4` is the equilibrium throughput default; combined with `clear_on_submit=True` it resets each week so the player consciously decides.
   - `key="order_input"` MATCHES the read in `app.py::submit_order` (`st.session_state["order_input"]`). Without this exact key, the order won't flow through. Pitfall 2.
   - `on_click=on_submit` wires the submit to the callback parameter (app.py passes `submit_order` as `on_submit`).

```python
def test_submit_order_advances_one_week():
    at = AppTest.from_file("app.py", default_timeout=5).run()
    at.button[0].click(); at.run()                     # rules -> setup
    # Get to playing. Try form_submit_button if available; else best-effort.
    # Use direct session_state seeding as a robust fallback:
    if at.session_state.phase != "playing":
        # Bypass setup widget interaction — seed session_state directly.
        from beergame.engine.state import Role, new_game
        from beergame.ai import ShipmentAnchorAndAdjustAgent
        at.session_state.game = new_game(player_role=Role.RETAILER, seed=42)
        at.session_state.ai_agents = {
            r: ShipmentAnchorAndAdjustAgent()
            for r in Role if r != Role.RETAILER
        }
        at.session_state.phase = "playing"
        at.run()
    assert at.session_state.phase == "playing"
    assert at.session_state.game.week == 0

    # Set the order_input via session_state and trigger the submit_order
    # callback by simulating a click. If the form_submit_button is reachable
    # via at.button (it may or may not be; depends on 1.57.0 AppTest behavior),
    # use it; else invoke submit_order directly through the page rerun.
    at.session_state["order_input"] = 4
    # Find and click the "Advance week" button. In 1.57.0 AppTest, form submit
    # buttons may be at.button[-1] (last button on the page). Try that:
    try:
        at.button[-1].click()
        at.run()
    except Exception:
        # Fall back: invoke the callback directly. Import the function
        # from app.py module. This is testing the callback contract, not
        # the form submission path itself.
        import importlib
        app_mod = importlib.import_module("app")
        app_mod.submit_order()  # uses at.session_state via the underlying
                                 # streamlit testing harness
        at.run()

    assert at.session_state.game.week == 1
    assert at.session_state.phase == "playing"
    assert not at.exception
```
